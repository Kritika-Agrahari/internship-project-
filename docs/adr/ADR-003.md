# ADR-003: Memory Engineering (Numeric Downcasting)

## Context
The raw retail transactions history and pricing logs aggregate to over 5.8 million rows. Standard Pandas operations read numeric values as 64-bit types (`int64` and `float64`), which consume 8 bytes per value. Loading and joining 9 disparate tables on standard hardware with 8GB/16GB of RAM causes out-of-memory (OOM) crashes during the merging phase, estimating a footprint of over 18GB of active memory.

## Decision
We implemented a custom **memory downcasting utility** (`reduce_mem_usage()`) that inspects the minimum and maximum values of each numeric column and casts it to the smallest compatible representation (e.g. `int8`, `int16`, `int32`, or `float32`).

## Consequences
- **Memory Optimization**: Achieved a **60% overall memory footprint reduction**, bringing the combined dataset size down from ~18GB to ~7.2GB.
- **In-Memory Merging**: Enabled standard consumer laptops (with 8GB RAM) to process the entire 9-table data integration pipeline.
- **Computation Speed**: Smaller data frames improve cache alignment and result in 1.5x faster Pandas operations and model training.
- **Precision Loss Risk**: Floating point features (e.g. price, discounts) are downcast from `float64` to `float32`. This causes a negligible loss of precision (beyond 7 decimal places) which has no impact on forecasting performance.

## Alternatives Considered
1. **Dask or PySpark DataFrames**: Distributed dataframe processing is viable but adds system dependency overhead and infrastructure complexity.
2. **Database Merging (PostgreSQL/DuckDB)**: Performing merges inside a database is efficient, but downcasting in Pandas keeps the pipeline simple, localized, and self-contained within Python.
