# ADR-002: Validation Split Strategy (Chronological vs. Random)

## Context
When training machine learning models on time-sensitive datasets, evaluating them using random train/test splits (e.g. `train_test_split` with shuffling) leads to temporal data leakage. Features such as historical sales lags or rolling window averages would contain look-ahead bias, making the validation metrics artificially high and causing the model to perform poorly on real future data.

## Decision
We chose a **strict chronological (date-based) train/test split** where the final 90 days of the dataset are held out for testing, and the preceding historical data is used for training. Shuffling is strictly disabled.

## Consequences
- **No Temporal Leakage**: The model only trains on past data and predicts the future, reflecting actual production usage.
- **Accurate Evaluation**: The test RMSE (7.6788) represents a true out-of-sample forecast accuracy.
- **Feature Robustness**: Lag features (e.g. lag_7, lag_14) are computed based on actual chronological history, preventing overlapping signals.
- **Data Volume Trade-off**: The training set is slightly smaller, but chronological splitting ensures the model generalization to seasonal patterns is validated correctly.

## Alternatives Considered
1. **Random K-Fold Cross-Validation**: Rejected because random shuffling mixes future and past data, leading to massive look-ahead bias and overfitting.
2. **Rolling Window Cross-Validation (Time Series Split)**: A solid alternative, but rejected for the baseline MVP due to the extra compute power required to train the model multiple times on expanding windows. This will be considered in the 3rd-year roadmap.
